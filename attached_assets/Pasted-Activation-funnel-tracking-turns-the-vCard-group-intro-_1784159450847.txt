Activation funnel tracking (turns the vCard/group-intro growth loop into something measurable). Right now a "first-ever message from a never-before-seen number in an existing thread" is a real event happening in your code (it's exactly when resolveGroupParticipants/threadParticipants creates a new user row inside a group) but nothing marks it as a distinct funnel stage — it looks identical to any other new user creation. Add a source: "cold_dm" | "group_add" (and maybe originThreadId) column on users, set at creation time, and log a lightweight activation_events table (userId, event: "added_to_group" | "first_reply" | "onboarding_complete", timestamp). This is maybe 20 lines across context.ts and one migration, and it's what turns "I think the vCard is working" into an actual invite→activate conversion number you can watch.
Delivery health as a first-class Operations metric, not just a log table. messageDeliveryLogTable already captures ERROR/FAILED/DELIVERED per send, but nothing rolls it up. Add a simple query + card on the Operations.tsxdashboard: delivery success rate per rolling 24h, broken out by thread, with anything trending down flagged visually. This is pure aggregation over data you're already writing — no new capture logic — and it's the earliest warning you'll get of number-reputation problems, which otherwise show up as silent, invisible churn.
A real opt-out path for bystanders, independent of you personally. Today, muting exists (MUTE_PATTERN) but it silences the bot per-participant — it doesn't answer "who is this number and how do I make it never contact me again" for someone who never asked to be in this experiment. Concretely: (a) add one static line to the disclosure DM and group intro with a link to a real privacy policy (even a single markdown page hosted at a stable URL is enough for now — the point is it exists and is linkable, not that it's exhaustive); (b) make the mute command's confirmation reply explicitly state what muting does and does not do ("I'll stay quiet in this thread, but I may still be reachable in others"); (c) add a "forget me" / "delete my data" phrase, deterministic like mute, that actually deletes the user's profile row and scrubs their messages — you already surface "what do you know about me?" via knowledge.ts, so the delete counterpart is a natural, small addition to the same module, and having it before your first non-friends user is much cheaper than retrofitting it after real conversations exist.
Per-thread cost tracking, since none currently exists. Every OpenAI call in engine.ts, etiquette.ts, onboarding.ts, groupIntro.ts, and privateInput.ts sets max_completion_tokens but nothing captures usage from the response back into the database. Add a costLog insert (threadId, tokens in/out, estimated cost, which module made the call) at each call site, or centralize it by wrapping your openai client once. Surfacing "$/active thread/week" on the Operations dashboard next to engagement numbers is what lets you catch a cost/engagement mismatch before it's a problem instead of after.
A lightweight "something's wrong" escape hatch for support, given you're the only support team. A deterministic phrase (like "contact support" or "this is broken", same pattern as MUTE_PATTERN) that flags the thread — sets a needsAttentionboolean and timestamp on threads, surfaced at the top of the Threads.tsx dashboard page — turns "I read every thread daily" from a manual habit into something you can't accidentally miss. Cheap, and it directly de-risks the solo-founder support load point.
A hook for the booking-automation decision, without building automation. Since the plan is human-assisted booking for hard reservations rather than API/scraping automation, add a concierge_assist_requested boolean to bookings (or a note field), set when the user's language signals it's a hard table ("can you actually get us in," "this place never has availability"). It doesn't need to do anything yet — but it gives you a queryable list of exactly which bookings would benefit from a human touch, which is the evidence you'd want before deciding whether to staff that tier at all.
